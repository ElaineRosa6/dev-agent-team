# 循环迭代研发 Agent 团队工作流

## 一、团队总览

### 1.1 角色矩阵

| 角色 | 核心使命 | 阶段归属 | 关键输出 |
|------|----------|----------|----------|
| **产品架构师** | 将业务需求转化为架构蓝图 | 规划阶段 | 演进式架构落地切片计划 |
| **安全切片规划师** | 将变更转化为安全可回滚的执行步骤 | 规划阶段 | 安全执行切片计划 |
| **代码审查专家** | 微观层面发现代码 Bug、安全漏洞、逻辑缺陷 | 审查阶段 | 四维度审查报告 |
| **架构审计师** | 宏观层面评估架构合理性、业务闭环、完成度 | 审查阶段 | 架构审计报告 |
| **自动修复工程师** | 按切片计划逐项核实并修复问题 | 修复阶段 | 修复摘要报告 |
| **运维管家** | 清理、记录、归档、迭代节奏管理 | 收尾阶段 | 工作记录、迭代总结 |

## 二、完整工作流

### 2.0 阶段零：迭代启动

**负责人**：运维管家

**触发条件**：新项目启动 / 上一轮迭代结束 / 新需求输入

**工作内容**：

1. 运维管家读取上一轮迭代总结（如有），恢复上下文
2. 确认本轮迭代的目标和范围
3. 准备输入材料：需求文档、遗留问题清单、架构文档

**输出**：

```markdown
## 迭代 N 启动

### 迭代目标
[本轮要达成的核心目标]

### 输入材料
- 需求文档：[路径]
- 上一轮迭代总结：[路径]
- 遗留问题清单：[路径]

### 参与角色
- 产品架构师：[是否参与]
- 安全切片规划师：[是否参与]
- 代码审查专家：[是否参与]
- 架构审计师：[是否参与]
- 自动修复工程师：[是否参与]
```

---

### 2.1 阶段一：架构规划

**负责人**：产品架构师 → 安全切片规划师

**触发条件**：新项目 / 新需求 / 重构计划

#### 步骤 1.1：产品架构师 — 演进式架构设计

**输入**：业务需求文档、现有系统上下文、技术栈约束

**工作内容**：

1. 识别 MVP 与领域边界
2. 定义 API 契约
3. 设计外部依赖防腐层
4. 规划可观测性与兜底策略
5. 输出演进式架构落地切片计划（阶段 0 → 阶段 3）

**输出**：演进式架构落地切片计划

**关键检查点**：

- [ ] MVP 链路是否可在无第三方环境下跑通？
- [ ] API 契约是否已定义？
- [ ] 外部依赖是否都有防腐层？

#### 步骤 1.2：安全切片规划师 — 安全执行切片

**输入**：架构落地切片计划、变更计划

**工作内容**：

1. 变更类型定性（新增/修改/重构/迁移）
2. 依赖链路梳理（数据层 → 服务层 → 接口层 → 前端）
3. 兼容与过渡方案设计
4. 输出安全执行切片计划（步骤 1 → 步骤 N）

**输出**：安全执行切片计划

**关键检查点**：

- [ ] 是否存在循环依赖？
- [ ] 所有破坏性动作是否有缓冲？
- [ ] 每个步骤是否可独立回滚？

**阶段一门控**：架构落地切片计划和安全执行切片计划必须经确认后，才能进入下一阶段。

---

### 2.2 阶段二：代码实现

**负责人**：开发团队（非 Agent 团队成员，但受切片计划约束）

**触发条件**：安全执行切片计划确认通过

**工作内容**：

1. 按安全执行切片计划的步骤顺序实现代码
2. 每个步骤完成后执行该步骤的验收标准
3. 验收通过后进入下一个步骤
4. 任何步骤验收失败，回滚该步骤并重新实现

**约束**：

- 严格按照切片计划的步骤顺序执行，不得跳步
- 每个步骤必须通过验收标准才能进入下一步
- 遇到切片计划未覆盖的情况，暂停并反馈给安全切片规划师

**输出**：实现代码 + 各步骤验收记录

---

### 2.3 阶段三：代码审查

**负责人**：代码审查专家 → 架构审计师

**触发条件**：代码实现完成 / 修复后回归审查

#### 步骤 3.1：代码审查专家 — 四维度深度审查

**输入**：待审查代码、技术栈信息、业务背景

**工作内容**：

1. **先看骨架**：分析目录结构和技术栈，提炼核心业务主流程
2. **顺藤摸瓜**：沿核心主流程检查代码，找出伪实现
3. **四维度扫描**：
   - 功能性 Bug（边界条件、异常处理、并发异步、类型状态、资源管理）
   - 安全性问题（注入攻击、认证授权、数据泄露、业务安全、依赖安全）
   - 逻辑与代码质量（代码坏味道、设计原则、性能隐患、可读性）
   - 业务逻辑问题（场景遗漏、数据一致性、时序问题）
4. 按严重程度排序输出审查报告

**输出**：四维度审查报告（严重 > 高 > 中 > 低）

**关键检查点**：

- [ ] 四个维度是否全部覆盖？
- [ ] 每个问题是否有具体文件路径和行号？
- [ ] 严重问题是否有修复代码片段？

#### 步骤 3.2：架构审计师 — 宏观层面审计

**输入**：项目源码、项目文档、代码审查报告、架构设计文档

**工作内容**：

1. **架构合理性评估**：模块划分、耦合关系、分层设计、可扩展性
2. **业务逻辑闭环诊断**：核心链路、异常流、前后端闭环
3. **项目完成度盘点**：完成度百分比、核心功能缺口、体验优化点
4. **代码质量与缺陷排查**：P0/P1/P2 级问题分类

**输出**：架构审计报告

**关键检查点**：

- [ ] 模块间是否存在循环依赖？
- [ ] 核心业务链路是否端到端打通？
- [ ] 是否存在 P0 级致命缺陷？

**阶段三门控**：审查报告和审计报告合并后，进入问题分类决策：

```
审查结果决策树：

存在问题？
├── 是 → 问题严重程度？
│   ├── P0（致命）→ 立即进入阶段四（修复）
│   ├── P1（高）  → 进入阶段四（修复）
│   ├── P2（中）  → 评估是否本轮修复或推迟
│   └── P3（低）  → 记录，推迟到后续迭代
└── 否 → 进入阶段五（收尾）
```

---

### 2.4 阶段四：安全修复

**负责人**：安全切片规划师 → 自动修复工程师

**触发条件**：审查/审计发现 P0/P1 级问题，或决定修复的 P2 级问题

#### 步骤 4.1：安全切片规划师 — 修复切片规划

**输入**：代码审查报告、架构审计报告

**工作内容**：

1. 汇总所有待修复问题
2. 识别问题间的依赖关系
3. 为破坏性修复设计兼容策略（防腐层/双写/Feature Toggle）
4. 输出修复执行切片计划

**输出**：修复执行切片计划

**关键检查点**：

- [ ] 修复顺序是否遵循依赖关系？
- [ ] 破坏性修复是否有兼容层？
- [ ] 每个修复步骤是否可独立回滚？

#### 步骤 4.2：自动修复工程师 — 分阶段修复执行

**输入**：修复执行切片计划、代码审查报告、架构审计报告、项目源码

**工作内容**：

1. **阶段零**：定位审查报告，探测项目环境
2. **阶段一**：高可信度快速修复（P1/P2 明确 Bug）
3. **阶段二**：需运行测试验证的修复（并发、状态机、边界条件）
4. **阶段三**：仅评估不修复（架构级问题）
5. **阶段四**：明确跳过（已修复、风格问题、需重写模块）
6. 运行构建/测试/lint 验证

**输出**：修复摘要报告 + 验证结果

**关键检查点**：

- [ ] 每个问题是否先核实再修复？
- [ ] 修复是否遵循最小改动原则？
- [ ] 修复后是否通过构建/测试验证？

#### 步骤 4.3：回归审查

**负责人**：代码审查专家

**工作内容**：

1. 对修复后的代码进行回归审查
2. 确认修复是否引入新问题
3. 确认原问题是否真正解决

**回归审查决策**：

```
回归审查结果：
├── 通过 → 进入阶段五（收尾）
├── 发现新问题 → 新问题严重程度？
│   ├── P0/P1 → 回到步骤 4.1（重新规划修复切片）
│   └── P2/P3 → 记录，进入阶段五
└── 原问题未解决 → 回到步骤 4.2（重新修复）
```

---

### 2.5 阶段五：迭代收尾

**负责人**：运维管家

**触发条件**：回归审查通过 / 决定推迟剩余问题

**工作内容**：

1. **垃圾文件清理**：扫描项目，列出清理清单，确认后执行
2. **工作记录**：记录本轮迭代的所有工作内容
3. **迭代总结**：汇总迭代成果、遗留问题、下一迭代计划
4. **文档归档**：按规范归档所有文档

**输出**：

- 清理清单（确认后执行）
- 工作记录
- 迭代总结
- 归档文档

**关键检查点**：

- [ ] 清理后项目是否仍可正常构建和测试？
- [ ] 工作记录是否完整？
- [ ] 遗留问题是否已记录并安排到后续迭代？
- [ ] 文档是否按规范归档？

### 3.2 迭代触发条件

| 触发场景 | 进入阶段 | 说明 |
|----------|----------|------|
| 新项目启动 | 阶段零 | 全新项目，从架构规划开始 |
| 新需求输入 | 阶段一 | 已有项目新增功能，从架构规划开始 |
| 回归审查未通过 | 阶段四 | 修复引入新问题或原问题未解决，重新修复 |
| 遗留问题处理 | 阶段一 | 上一轮推迟的问题，重新规划 |
| 代码优化/重构 | 阶段一 | 主动优化，从安全切片规划开始 |

### 3.3 迭代退出条件

| 退出场景 | 条件 | 后续动作 |
|----------|------|----------|
| 项目完成 | 所有功能实现、所有 P0/P1 问题修复、回归审查通过 | 归档所有文档，项目结项 |
| 迭代截止 | 达到迭代时间盒或工作量上限 | 汇总当前成果，遗留问题推至下一迭代 |
| 外部阻塞 | 依赖外部条件（第三方 API、环境等）无法继续 | 记录阻塞项，暂停迭代 |

### 3.4 迭代节奏建议

| 迭代类型 | 建议周期 | 重点 |
|----------|----------|------|
| MVP 迭代 | 1-2 周 | 快速验证核心业务闭环 |
| 功能迭代 | 1-2 周 | 填充功能、修复问题 |
| 稳定性迭代 | 1 周 | 集中修复 P0/P1 问题、补齐可观测性 |
| 重构迭代 | 2-3 周 | 架构优化、技术债务清理 |

## 五、异常处理与回退机制

### 5.1 阶段内回退

| 阶段 | 异常场景 | 回退策略 |
|------|----------|----------|
| 阶段一 | 架构设计发现不可行 | 回退到需求分析，重新定义 MVP |
| 阶段二 | 某步骤验收失败 | 回滚该步骤代码，重新实现 |
| 阶段三 | 审查发现 P0 问题 | 立即进入阶段四修复 |
| 阶段四 | 修复引入新问题 | 回滚该修复，重新规划切片 |
| 阶段四 | 回归审查未通过 | 回到步骤 4.1 重新规划 |
| 阶段五 | 清理后构建失败 | 回滚清理操作，排查原因 |

### 5.2 跨迭代回退

| 场景 | 回退策略 |
|------|----------|
| 上线后发现问题 | 紧急迭代：直接进入阶段四修复 |
| 架构决策错误 | 回退到阶段一重新规划，评估影响范围 |
| 外部依赖不可用 | 暂停当前迭代，启动 Mock 方案，进入下一迭代 |

### 5.3 紧急通道

当出现 P0 级线上问题时，可跳过常规流程，直接走紧急通道：

```
P0 紧急通道：

发现问题 → 代码审查专家(快速定位) → 安全切片规划师(紧急切片) → 自动修复工程师(紧急修复) → 回归审查 → 上线
```

## 六、质量门控

### 6.1 各阶段门控标准

| 阶段 | 门控标准 | 不通过处理 |
|------|----------|------------|
| 阶段一 | 架构计划 + 切片计划均确认通过 | 重新规划 |
| 阶段二 | 所有步骤验收通过 | 回滚失败步骤 |
| 阶段三 | 无 P0 问题 | 进入修复阶段 |
| 阶段四 | 回归审查通过 | 重新修复 |
| 阶段五 | 构建通过 + 测试通过 + 文档归档 | 修复问题后重新验证 |

### 6.2 迭代完成标准

一轮迭代被认为"完成"需满足以下全部条件：

- [ ] 迭代目标中定义的所有功能已实现
- [ ] 代码审查无 P0 问题
- [ ] 架构审计无 P0 问题
- [ ] 所有 P0/P1 问题已修复且回归审查通过
- [ ] 构建通过
- [ ] 测试通过
- [ ] 文档已归档
- [ ] 迭代总结已输出

## 七、快速参考

### 7.1 角色速查

| 我想... | 找谁 |
|---------|------|
| 设计新项目架构 | 产品架构师 |
| 制定安全的变更计划 | 安全切片规划师 |
| 审查代码 Bug 和安全漏洞 | 代码审查专家 |
| 评估架构合理性和完成度 | 架构审计师 |
| 修复审查发现的问题 | 自动修复工程师 |
| 清理文件、记录工作、归档文档 | 运维管家 |

### 7.2 阶段速查

| 阶段 | 核心问题 | 关键角色 |
|------|----------|----------|
| 阶段零 | 我们要做什么？ | 运维管家 |
| 阶段一 | 怎么做才安全？ | 产品架构师、安全切片规划师 |
| 阶段二 | 把代码写出来 | 开发团队 |
| 阶段三 | 代码有没有问题？ | 代码审查专家、架构审计师 |
| 阶段四 | 把问题修掉 | 安全切片规划师、自动修复工程师 |
| 阶段五 | 收尾和记录 | 运维管家 |

### 7.3 关键原则速查

| 原则 | 说明 |
|------|------|
| 禁止直接生成最终代码 | 架构师和规划师只输出计划，不输出代码 |
| 契约先行 | 先定义 API 契约，再实现逻辑 |
| 防腐层设计 | 破坏性修改必须有兼容层 |
| 最小改动 | 修复遵循最小改动原则，不引入新依赖 |
| 先核实再动手 | 修复前先确认问题是否真实存在 |
| 可回滚性 | 每个步骤必须可独立回滚 |
| 端到端验收 | 验收标准是业务场景验证，不是"代码不报错" |
