# Sprint Team 深度问答 — 6 个关键问题与回答

> 记录时间：2026-05-21
> 目的：记录对项目核心设计的深度质疑与回答，作为后续实施和迭代的启发文件。

---

## Q1：阶段二"代码实现"谁来干？LLM 还是人类？

### 问题

文档写死阶段二由"开发团队（非 Agent）"负责。

- 设想的"开发团队"是纯人类，还是也可以是另一个 LLM/Agent？
- 如果未来想逐步自动化阶段二，是让自动修复工程师慢慢吃掉实现工作，还是单独造一个"实现工程师"角色，还是走"代码沙盒"路线？

### 回答

**不应该让自动修复工程师做新功能实现**。它的 Prompt 是"按切片计划修复问题"，行为模式是诊断性的，不是创造性的。让一个角色同时做"修已知问题"和"创造未知功能"会混淆两种心智。

**更合理的路径：单独造"实现工程师"角色，走代码沙盒路线**。

```
LLM 生成代码 → 在沙盒里跑测试/编译/lint → 全部通过才留下 → 任何一个失败就打回
```

**阶段演进路线**：

| 阶段 | 模式 |
|------|------|
| 现在 | 人类实现（100%） |
| 过渡期 | LLM 生成 → 人 Review → 合入 |
| 目标 | LLM 生成 → 测试/编译锁死 → 自动合入（仅限低风险变更） |

**但现在不是做这个的时候**。先把防御基建做好（超时、trace、白名单），再接 LLM 写代码。没有降落伞之前不要开飞机。

## Q2：自动修复工程师的边界 — 它到底能改多深？

### 问题

- 哪些级别的修复可以放心让 LLM 自动改？只限 P2 以下？还是也会碰 P1？
- "需要测试验证的修复"是真正跑测试，还是只让 LLM 自己评估风险？
- 如果没有真正接入测试/编译器，那本质上还是"LLM 自我反思"，在逻辑上不可靠。

### 回答

**现状**：文档说会"运行构建/测试/lint 验证"，但实际代码里没有真正的测试接入。`output-validator.py` 只检查文件内容，不执行任何验证。

**哪些修复可以交给 LLM**：

| 级别 | 类型 | 是否可交给 LLM | 条件 |
|------|------|--------------|------|
| P0 | 安全漏洞（SQL 注入、XSS） | **不可以** | 需要人确认 |
| P1 | 明确 Bug（空指针、边界条件） | **可以** | 需要测试覆盖 |
| P2 | 代码坏味道（超长函数、命名） | **可以** | 低风险 |

**核心判断**：没有真正接入测试/编译器，自动修复工程师本质上就是"LLM 自我反思"——自己出题自己答，这是不可靠的。

**必须接入的闭环**：

```
1. 记录修复前的测试状态（多少个 pass/fail）
2. 执行最小改动
3. 重新跑测试 → 对比结果
4. 测试通过 + lint 通过 → 记录成功
5. 测试失败 → 回滚修复，标注失败原因
```

没有这套闭环，这个角色就是个文档角色，不是工程角色。

## Q3：Hooks + Scripts 是门卫还是可观测性主角？

### 问题

目前 Hooks 偏"拦截/校验/状态同步"，没有监控/指标部分。有没有计划让 Hooks 同时承担"翻车指标采集"？数据是只写日志还是做成仪表盘？

### 回答

**现状**：纯门卫，没有承担可观测性。

- Guard 拦截日志写到了 stderr，但没有持久化
- `output-validator.py` 的 `.validation-log.txt` 追加写入，不结构化
- `sprint-state-sync.py` 只更新 JSON，不记录历史

**应该怎么做**：让每个 Hook 同时输出结构化事件。

```json
{
  "timestamp": "2026-05-21T10:30:00Z",
  "hook": "permission-guard",
  "action": "blocked",
  "role": "product-architect",
  "phase": "one_architect",
  "attempted_path": "src/main.py",
  "reason": "角色不允许写入非 docs/architecture/ 目录"
}
```

数据写入 `sprints/logs/guard/sprint-{N}-guard.jsonl`，每轮迭代结束自动生成指标报告：

```
sprint-003 翻车指标快照：
  - Guard 拦截次数：12（vs 上轮 8，+50%）
  - 输出校验失败：3（vs 上轮 1）
  - 阶段重试：1（code-reviewer 超时）
  - 人工介入：2 次
  - 健康分：72（vs 上轮 85，↓13）
```

**不是先做仪表盘，先做到每轮自动生成报告**。仪表盘是后面的事。

## Q4：工作流本身的逻辑正确性由谁保证？

### 问题

工作流靠人肉审 + 文档约定来维护，还是有形式化/可执行的方式描述？`run_sprint.py` 里是不是已经有阶段转换 + 门控条件的硬逻辑？未来调整工作流是只改文档，还是同步改脚本 + 测试？

### 回答

**关键发现**：`run_sprint.py` 在 `.gitignore` 里，**还没有实际实现**。工作流逻辑完全靠文档约定。

这意味着：
- 改了 Prompt 但忘了更新状态机字段名 → 没有检测
- 加了新角色但忘了加权限 → 运行时才发现
- 阶段顺序改了但文档没同步 → 文档和实际不一致

现在的工作流是"写得很好的文档"，不是"有逻辑保证的执行规范"。

**应该怎么做**：把状态机定义抽成独立的配置文件。

```json
{
  "phases": [
    { "id": "zero", "role": "devops-manager", "gate": false, "required_artifacts": ["iteration-start.md"] },
    { "id": "one_architect", "role": "product-architect", "gate": true, "required_artifacts": ["architecture-plan.md"] },
    { "id": "one_planner", "role": "slice-planner", "gate": true, "depends_on": ["one_architect"], "required_artifacts": ["execution-plan.md"] }
  ],
  "decisions": [
    {
      "after_phase": "three_audit",
      "condition": "has_p0_or_p1_issues",
      "if_true": "four_fix_plan",
      "if_false": "five_summary"
    }
  ]
}
```

然后用代码验证：阶段转换是否符合 DAG、必需产物是否生成、门控条件是否满足。

**未来加角色/阶段**：改配置文件 + 写测试用例，不是改文档 + 祈祷。

## Q5：安全切片规划师 — 最逻辑密集的角色，LLM 能胜任吗？

### 问题

安全切片规划师要考虑依赖链、兼容策略、回滚方案，这些都是高度逻辑的活。实际有多依赖 LLM？有没有考虑把依赖分析/回滚方案抽成独立工具，让 LLM 只负责"把人类语言翻译成工具输入"？

### 回答

**是的，这个角色最容易翻车**。它要求的能力（依赖分析、回滚方案设计、兼容性判断）是纯逻辑活，LLM 做这个最不可靠。

**现状**：纯 Prompt 驱动。给架构计划 → LLM 输出切片计划 → 人看一眼确认。没有依赖分析工具、没有回滚验证。

**应该拆成两层**：

```
Layer 1: 传统工具做逻辑推演（不会翻车）
  - 依赖分析脚本：AST 解析 / import 分析 → 生成依赖图 → 检测循环依赖
  - 回滚方案模板引擎：根据变更类型，自动生成标准回滚模板
  - 兼容性检查器：对比修改前后的 API/schema，检测破坏性变更

Layer 2: LLM 做意图翻译（翻车了好修）
  - 用户的模糊意图 → 翻译成结构化输入给工具
  - 工具的输出 → 翻译成人类可读的切片计划
```

**核心原则**：让 LLM 做"意图翻译"，让传统工具做"逻辑推演"。

逻辑推演交给工具 → 这部分不会翻车。
意图翻译交给 LLM → 这部分翻车了也好修。

## Q6：紧急通道和跨迭代回退 — 有多硬？

### 问题

紧急通道是文档约定 + 人手动切换，还是已经在代码里做了模式切换？跨迭代回退有没有版本标记 + 回滚脚本，还是靠人肉 + 文档追述？

### 回答

**现状**：文档里有规则，代码里完全没有实现。

`run_sprint.py` 还没写，`sprint-state.json` 没有"紧急模式"字段，没有回滚脚本。完全是"约定俗成 + 人肉执行"。

如果用户说"走紧急通道"，系统不会跳过不该跳的步骤、不会自动标记紧急模式、不会在事后强制补回被跳过的审查、不会防止紧急通道被滥用。

**应该做成硬逻辑**：

```json
{
  "mode": "normal | emergency",
  "emergency_activated_at": null,
  "emergency_reason": null,
  "skipped_phases": [],
  "post_emergency_review_required": true
}
```

```python
def activate_emergency(reason: str):
    state["mode"] = "emergency"
    state["skipped_phases"] = ["one_architect", "one_planner"]
    # 但不能跳过审查和回归
    # 紧急通道 = 跳过规划 → 直接修复 → 必须回归审查 → 必须补文档

def post_emergency_check():
    if state["mode"] == "emergency" and not review_done():
        raise EmergencyReviewRequired("紧急修复后必须补代码审查和架构审计")
```

跨迭代回退应该有版本标记：

```json
{
  "sprint_id": "sprint-003",
  "git_commit": "abc123",
  "architecture_hash": "sha256:...",
  "can_rollback_to": true
}
```

回退不是"人肉追述"，是 `git checkout <commit>` + 恢复当时的架构计划。

## 总结

6 个问题的共同答案：**文档很完善，代码没跟上**。

系统目前的状态是"设计完成了 80%，实现了 20%"。Guard 脚本和权限配置是实际可运行的部分，但编排引擎、测试接入、可观测性、紧急通道这些硬核部分还没落地。

**这不是坏事**——设计先行避免了乱写代码。但下一步应该：

1. 先把 `run_sprint.py` 实现（状态机驱动的执行引擎）
2. 同步做防御基建（超时、trace、白名单 Guard）
3. 再接 LLM 写代码的功能

**顺序不能反**。
