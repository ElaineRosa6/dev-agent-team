# Sprint Team 项目分析与改进计划

## 核心前提

大模型和 Agent 必然翻车，这是系统特性而非 Bug。所有设计基于"翻车也翻不死"原则：防御优于修补、边界外置、兜底降级、白名单拦截、超时重试、全链路 trace、强制迭代比例。

## 现状

6 角色多 Agent 编排系统，5 阶段工作流（迭代启动→架构规划→代码实现→代码审查→迭代收尾）。
核心文件：`sprints/sprint-state.json`（状态机）、`sprint-team-plugin/scripts/run_sprint.py`（编排引擎）、`sprints/prompts/*.md`（角色 Prompt）。

## 发现的 4 个功能缺口

1. **无项目感知层** — Agent 不会自动扫描项目理解现状
2. **无技术债务追踪** — P2/P3 问题跨迭代无限推迟
3. **无 LLM 错误累积防护** — 无基线快照、无模型错误检测、无健康评分
4. **无棕地入口** — 不支持"已有项目 + 问题清单"启动

## 8 个脆弱点（"必然翻车"审视）

1. 无超时/重试/降级 → hang 时静默死亡
2. Guard 是黑名单模式 → 注定遗漏可绕过
3. 校验只检查存在性 → 伪实现和废话报告通过
4. 状态机无失败状态 → 中途崩溃后状态不一致
5. 无 trace 链路 → 翻车后无法复现和根因分析
6. 角色间无会话隔离 → 一个跑飞全盘跑飞
7. Prompt 散落无版本管理 → 改了不知道、无法回退
8. 无强制迭代比例 → 债务无限堆积

## 9 项技术债务（按紧迫性）

P0: D-001 Guard 改白名单 / D-002 校验增强 / D-003 超时重试降级
P1: D-004 trace 链路 / D-005 状态机失败恢复 / D-006 Prompt 版本化 / D-007 会话隔离
P2: D-008 迭代比例约束 / D-009 Prompt 去重

还债计划：一个稳定性迭代周期（6 天）分 3 个 Phase 完成。

## 计划新增 8 个组件

**P0**：项目感知层、技术债务账本、棕地入口流程
**P1**：基线快照系统、模型错误猎犬、健康评分系统、问题切入点分析、棕地审查入口

## 8 个翻车指标

人工介入率、幻觉拦截率、工具失败率、降级触发率、阶段重试率、债务超期率、基线漂移度、伪实现检出率

## 约束

不改变现有工作流，状态向后兼容，防御优先，白名单优于黑名单。
